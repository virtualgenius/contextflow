Drag-direction options (interactive)

Two gestures in each panel. Hover a context to reveal arrow stubs on its four sides; drag any stub to another context to draw a relationship line (same affordance as connection-ux-mockup.html). Drag a context's body to move it (and in Options B and C, dropping one context overlapping the other creates or switches to Shared Kernel).

Background: PLAN-two-sided-relationships.md, DDD_RELATIONSHIP_PATTERNS.md. P1 was set in connection-ux-mockup.html (GH #22). Shared-Kernel overlap is tracked in bead contextflow-cqi.

Vocabulary (Evans, DDD Reference pp. 37-38). Contexts have three levels of influence on each other. Upstream/Downstream: asymmetric, the upstream's decisions propagate to the downstream. Mutually Dependent: bidirectional, success or failure of one affects the other (Partnership, Shared Kernel). Free: "direction, success or failure of development work in other contexts has little affect on delivery."
Influence type is internal vocabulary, not a picker control. The app's domain model knows which patterns belong to which influence type (Customer-Supplier → U/D; Partnership, Shared Kernel → Mutually Dependent) and uses that for rendering, validation, and grouping. Users pick a concrete pattern (or per-side roles), not an influence type. Free isn't a pickable value either, since it means "no relationship exists" — to express Free, the user removes the relationship.
Upstream/Downstream
Connecting line with an arrow head pointing at upstream. Per-side roles (OHS/PL, Conformist/ACL) optionally characterize each end.
Mutually Dependent → Partnership
Connecting line, no arrow head.
Mutually Dependent → Shared Kernel
Contexts overlap. The shared kernel renders in the intersection (hatched). No connecting line. Created by dragging contexts to overlap, or by picking it from the picker (which snaps them to overlap). (Bead contextflow-cqi.)
Free
No connecting line. Free means "deliberately not integrating" or "not related." Document on the context itself if needed, not as a relationship. Separate Ways is one named instance of Free.

Direction control alternatives

Six ways to display and flip direction in the picker

Click any swap/flip control to flip direction. All six mockups share state so the comparison stays apples-to-apples.
Option α (current)
Pill + flip button
Compact and explicit. Two things to read (the display and the button). Helper text needed to explain what "→" means.
Option β
Whole pill is the toggle
Single click target. Hover reveals "click to flip." Discoverability concern: users may not know the pill is clickable.
Option γ
Mini-diagram with swap icon
Most legible at a glance; arrow direction (and yellow "upstream" highlight) reflect the relationship. Contexts stay in their positions; the swap button flips the arrow.
Option η
Mini-diagram, arrow is the swap control
Like γ but no separate swap button. Single-click the arrow to flip its direction (contexts stay in their positions). Hover state on the arrow signals it's clickable.
Option δ (recommended)
Integrated into per-side rows
Per-side rows already name upstream and downstream contexts; the swap button goes where the user's eyes already are. No separate Direction section needed.
Option ε
On-canvas only (no picker control)
Picker just reports the direction. To flip, double-click the arrow line in the mini-canvas — the arrow direction reverses, contexts stay in their positions. Keeps picker focused on patterns; canvas gesture is hidden.
Option ζ
Two-option radio
Reads like prose, zero ambiguity. Takes more vertical space; verbose if both context names are long.
Implications for the data model and gestures. Two patterns leave Relationship entirely:
Separate Ways / Free
No Relationship object exists for the pair. The existing pattern: 'separate-ways' migration becomes a deletion.
Shared Kernel
Modeled as a separate entity (per bead contextflow-cqi). Two creation gestures: drag a context onto the other, or pick Shared Kernel in the picker (which snaps them to overlap). The existing pattern: 'shared-kernel' migration becomes "create a SharedKernel; delete the relationship."
Partnership
Stays as a standalone Relationship.pattern value (Mutually Dependent, no arrow head).
Customer-Supplier
Stays as a standalone Relationship.pattern value (Upstream/Downstream).
Per-side roles
OHS, Published Language, Conformist, ACL, applicable only when influence is Upstream/Downstream.
