ContextFlow / Add Context UX

Adding contexts: double-click + semantic arrows (mouse and keyboard)

One model, two input paths. Double-click empty canvas to seed a context with an inline name. Then grow the map with four semantic arrows that mean the same thing whether you click them on hover or press them on the keyboard: ↑ upstream provider, ↓ downstream consumer, ← partnership peer, → shared-kernel peer. Mouse and keyboard call the identical action, so muscle memory transfers between them.

Notes

Decisions

Shared kernel is the overlap, not an edge. In this app, two contexts overlapping is a Shared Kernel (dragging them apart auto-converts to Partnership). So the right arrow drops the new context partially overlapping the source rather than spaced apart with a connector. Partnership (left) and Shared Kernel (right) are then the same relationship at two overlap distances: adjacent-with-edge vs overlapping.
The four arrows are settled. Up = upstream, down = downstream, left = partnership, right = shared kernel. Separate ways and conformist were considered for the right slot and set aside.
First context vs. subsequent. Double-click (or N) creates a standalone context with no edge. Arrows always create a context related to the current one, so the relationship is captured at creation time instead of as a separate second step.
Directionality follows the codebase convention. Upstream provides, downstream consumes; the rendered edge direction matches the existing relationship-drawing convention rather than introducing a new one.
Plain arrow keys need an override. React Flow uses arrow keys for selection nudge and pan. The add-context binding only fires when a context is selected and no name input is focused; otherwise arrows fall through to default canvas behavior.
Collision and layout. If the target spot is occupied, offset further out (and pan if needed) rather than overlapping silently.
Toolbar Add Context stays as a discoverable fallback and for keyboard-only users who haven't learned the arrows yet. It lands at viewport center with the inline rename pre-focused, same end state as N.
