Strategic View: Icon Nodes Instead of Boxes

The Strategic View currently reuses the Flow View's large box nodes. On a Wardley map that fights the medium: boxes dominate the canvas, hide the value chain lines, and read as "containers" rather than positioned components. This mockup follows the visual vocabulary Susanne Kaiser uses in Architecture for Flow: small icon nodes for every element type, with domain classification, build/buy/outsource decisions, and Team Topologies overlays. The example is the online school from her InfoQ article and Architecture for Flow talk, using her bounded context decomposition.

Try it: drag nodes (edges, group hulls, and the evolution band highlight all follow), scroll or use the slider to zoom (decision tags hide first, then labels, mirroring the app's overlay zoom cutoffs), hover or click a node to trace its value chain two hops out, hover a group hull to highlight exactly its members, toggle system groups and team overlays independently, and use the dense map to stress-test label crowding. The teal and fuchsia hulls are Value Stream view groups reused as systems: each mixes bounded contexts with the skills, knowledge, and components that system depends on. Value chain reads top to bottom: users, needs, bounded contexts, infrastructure components. Bounded contexts are Kaiser's: content creation, class management, course studies, learning support (core); student evaluation, notification handling (supporting); identity & access (generic). Each gets its own stream-aligned team boundary. The dashed red arrow and inertia bar are Wardley-canon movement annotations.

What Changes vs. Today

Scope notes
  • Contexts shrink from 160px boxes to 40px glyphs (other elements 32px) in Strategic View only. Flow and Distillation views keep the box nodes.
  • Five new element types: capability, infrastructure component, skill/practice, knowledge, and expertise (the latter three are Wardley's practice/knowledge component types). All need the full Yjs round-trip (types, schema, sync, mutations) and analytics events. Open question: separate entities vs one "map element" type with a kind field. Capability is optional in Kaiser's method; needs may connect directly to contexts.
  • Decision tags and movement annotations (build/buy/outsource, evolution arrow, inertia) are new per-element fields, surfaced as toggleable overlays. The movement annotations are Wardley canon rather than Kaiser-specific.
  • Team overlays build on the existing teamId assignment and Team Topologies types, with one stream-aligned boundary per bounded context; this is the "team boundary overlay" quick win from the Susanne Kaiser feature ideas doc. Off by default to keep the base map quiet.
  • System groups reuse the existing Groups feature from the Value Stream view, rendered as the same organic hulls. Group membership widens beyond contexts to the new element types, so one group can show a whole system: its contexts plus the skills, knowledge, and components it depends on. Groups and team overlays are independent toggles, which is a small step toward Kaiser's overlay-multiple-perspectives idea.
  • Not drawn yet: Team Topologies interaction modes (collaboration, X-as-a-Service, facilitating) from her canvas. Her as-is to to-be landscape evolution is already covered by ContextFlow's temporal keyframes.
  • Icon and color choices reuse the inspector vocabulary (sparkles/wrench/package, amber/blue/slate) so the map and inspector tell one story.
